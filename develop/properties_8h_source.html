  381 * # UI declarative data binding {#UI_declarative_data_binding}
  382 * <b aui-src="aui.uitests/tests/UIDataBindingTest.cpp#L476"></b>
  383 * As said earlier, \c let syntax is a little bit clunky and requires extra boilerplate code to set up.
  384 *
  385 * Here's where declarative syntax comes into play. The logic behind the syntax is the same as in
  386 * `AObject::connect`/`AObject::biConnect` (for ease of replacement/understanding).
  387 *
  388 * Declarative syntax uses `&` and `&&` operators to set up connections. These were chosen intentionally: `&&` resembles
  389 * chain, so we "chaining view and property up".
  390 *
  391 * - `&` sets up one-directional connection (`AObject::connect`).
  392 * - `&&` sets up bidirectional connection (`AObject::biConnect`).
  393 *
  394 * Also, `>` operator (resembles arrow) is used to specify the destination slot.
  395 *
  396 * The example below is essentially the same as @ref "UIDataBindingTest_Label_via_let" but uses declarative connection set up syntax.
  397 *
  398 * ## Label via declarative {#UIDataBindingTest_Label_via_declarative}
  399 * <b aui-src="aui.uitests/tests/UIDataBindingTest.cpp#L491"></b>
  400 * Use `&` and `>` expression to connect the model's username property to the label's @ref ALabel::text "text"
  401 * property.
  402 * @code{cpp}
  403 *    using namespace declarative;
  404 *    struct User {
  405 *        AProperty<AString> name;
  406 *    };
  407 *
  408 *    auto user = aui::ptr::manage(new User { .name = "Roza" });
  409 *
  410 *    class MyWindow: public AWindow {
  411 *    public:
  412 *        MyWindow(const _<User>& user) {
  413 *            setContents(Centered {
  414 *              _new<ALabel>() & user->name > &ALabel::text
  415 *            });
  416 *        }
  417 *    };
  418 *    auto window = _new<MyWindow>(user);
  419 *    window->show();
  420 * @endcode
  421 *
  422 * ![](imgs/UIDataBindingTest.Label_via_declarative_1.png)
  423 * Note that the label already displays the value stored in User.
  424 *
  425 * Let's change the name:
  426 * @code{cpp}
  427 * user->name = "Vasil";
  428 * @endcode
  429 *
  430 * ![](imgs/UIDataBindingTest.Label_via_declarative_2.png)
  431 * In this example, we've achieved the same intuitive behaviour of data binding of `user->name` (like in
  432 * @ref "UIDataBindingTest_Label_via_let" example) but using declarative syntax. The logic behind `&` is almost the same as with `let`
  433 * and `AObject::connect` so projection use cases can be adapted in a similar manner.

  435 * ## ADataBindingDefault for omitting view property {#UIDataBindingTest_ADataBindingDefault_for_omitting_view_property}

  473 * ## ADataBindingDefault strong type propagation {#UIDataBindingTest_ADataBindingDefault_strong_type_propagation}
  474 * <b aui-src="aui.uitests/tests/UIDataBindingTest.cpp#L619"></b>
  475 * Think of `ADataBindingDefault` as we're not only connecting properties to properties, but also creating a
  476 * "property to view" relationship. This philosophy covers the following scenario.
  477 *
  478 * In AUI, there's aui::ranged_number template which stores valid value range right inside the type:
  479 * @code{cpp}
  480 * struct User {
  481 *     AProperty<aui::ranged_number<int, 1, 99>> age;
  482 * };
  483 * @endcode
  484 *
  485 *
  486 * These strong types can be used to propagate their traits on views, i.e., ANumberPicker. When using declarative
  487 * syntax, the property system calls `ADataBindingDefault::setup` to apply some extra traits of the bound value on
  488 * the view. Here's an abstract on how `ANumberPicker` defines specialization of `ADataBingingDefault` with
  489 * `aui::ranged_number`:
  490 * @code{cpp}
  491 * /* PREDEFINED! You don't need to define it! This listing is an example */
  492 * template <aui::arithmetic UnderlyingType, auto min, auto max>
  493 * struct ADataBindingDefault<ANumberPicker, aui::ranged_number<UnderlyingType, min, max>> {
  494 * public:
  495 *     static auto property(const _<ANumberPicker>& view) {
  496 *         return view->value();
  497 *     }
  498 *     static void setup(const _<ANumberPicker>& view) {
  499 *         view->setMin(aui::ranged_number<UnderlyingType, min, max>::MIN);
  500 *         view->setMax(aui::ranged_number<UnderlyingType, min, max>::MAX);
  501 *     }
  502 *     // ...
  503 * };
  504 * @endcode
  505 *
  506 * As you can see, this specialization pulls the min and max values from `aui::ranged_number` type and sets them
  507 * to `ANumberPicker`. This way `ANumberPicker` finds out the valid range of values by simply being bound to value
  508 * that has constraints encoded inside its type.
  509 * @code{cpp}
  510 * _new<ANumberPicker>() && user->age,
  511 * @endcode
  512 *
  513 * @note
  514 * We're using `operator&&` here to set up bidirectional connection. For more info, go to
  515 * @ref "UIDataBindingTest_Declarative_bidirectional_connection".
  516 *
  517 *
  518 * By creating this connection, we've done a little bit more. We've set ANumberPicker::setMin and
  519 * ANumberPicker::setMax as well:
  520 * @code{cpp}
  521 * EXPECT_EQ(numberPicker->getMin(), 1);
  522 * EXPECT_EQ(numberPicker->getMax(), 99);
  523 * @endcode
  524 *
  525 *
  526 * This example demonstrates how to use declarative binding to propagate strong types. `aui::ranged_number`
  527 * propagates its constraints on `ANumberPicker` thanks to `ADataBindingDefault` specialization.
  528 *

  567 * ## Declarative bidirectional connection {#UIDataBindingTest_Declarative_bidirectional_connection}
  568 * <b aui-src="aui.uitests/tests/UIDataBindingTest.cpp#L822"></b>
  569 * In previous examples, we've used `&` to make one directional (one sided) connection. This is
  570 * perfectly enough for ALabel because it cannot be changed by user.
  571 *
  572 * In some cases, you might want to use property-to-property as it's bidirectional. It's used for populating view

  574 *
  575 * For this example, let's use ATextField instead of ALabel as it's an editable view. In this case, we'd want to use
  576 * `&&` because we do want `user->name` to be aware of changes of the view.
  577 * @code{cpp}
  578 * _new<ATextField>() && user->name
  579 * @endcode
  580 *
  581 *

  585 * Let's change the name programmatically:
  586 * @code{cpp}
  587 * user->name = "Vasil";
  588 * @endcode
  589 *
  590 *
  591 * ATextField will respond:
  592 * ![](imgs/UIDataBindingTest.Declarative_bidirectional_connection_2.png)
  593 *
  594 * If the user changes the value from UI, these changes will reflect on `user->model` as well:

  599 *
  600 *
  601 * This way we've set up bidirectional projection via `&&` which makes `user->name` aware of UI
